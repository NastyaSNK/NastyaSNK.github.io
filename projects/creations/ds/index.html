Кластеризация тем и заданий курсов подготовки к егэ

Цель инструмента: Определить какие темы/задания решаются более успешно, какие - менее успешно.

Рис. 1 Схема работы итогового решения

Python использовался для отправки SQL запрос к системе-источнику и проведения кластеризации, PowerBi для визуализации результатов. Автоматизировался запуск скриптов через планировщик задач.

В данном демо кластеризация выполнена на 2 показателях:

для тем:

  • доля верных тестов от плана
  • доля верных тестов из решеных

для тестов:

  • решили/должны были решать(план) учеников
  • решили верно/план учеников

  • задач в теме по плану

для тем:

Прогноз ухода ученика с курса

В компании была потребность до совершения расторжения договора об оказании услуг знать об этом, чтобы была возмножсть проактивно повлиять на решение клиента и уменьшить значение показателя оттока.
Это нужно делать так как:

  • Потеря клиента -критична, это может повлиять на приток клиентов в будущем
  • Привлечение нового клиента стоит дороже, чем удержание
  • Согласно американской компании ReSci, уменьшение оттока клиентов на 5% может повлечь за собой увеличение прибыли до двух раз

Для удовлетворения потребностей был разработан “Инструмент прогноза расторжений”.

Рис.1 Этапы разработки и эксплуатации

При разработке инструмента стояли цели:
1️⃣ Анализ интересующего сегмента клиентов
2️⃣ Автоматический выбор наиболее значимых факторов оттока в выбранном сегменте
3️⃣ Визуализация данных по оттоку
4️⃣ Построение интерпретируемых моделей оттока

В MVP 1.0 вошла модели прогнозирования вероятности оттока, которая работала для отдельных продуктовых сегментов. Для модели использовались данные из нескольких источников (CRM,LMS), основной инструмент предобработки - Jupyter Notebook с Python и SQL скриптом.

Тк % склонных к оттоку клиентов значительно меньше активных клиентов, для борьбы с дисбалансом классов использовалась метод Smote бибилиотеки imblearn.
imbalanced-learn(imblearn) — это Python библиотека для борьбы с проблемами несбалансированных наборов данных.

Рис.2 Библиотека imblearn overview

Предобработанные данные для прогноза поступали в модель. Прогноз формировался в csv файле, который содержал инфо о клиентах, степени риска расторжения и доп информацию, которая может помочь в отработке. Отсутствие форонтенда у системы объяснялось тем, что проект находился в стадии эксперимента, осуществлялась проверка, какую точность может обеспечить модель на практике и будет ли иметь статистически значимый эффект.

Рис.3 Схема работы модели

Для решения задачи классификации лучше всего сработал градиентный бустинг. Модель достигла отличных показателей на валидации. При тестировании на реальных данных упала ее точность(precision), несмотря на это под проверку попадало 16% от общего количества договоров, что позволяло уточнить направленность маркетинговых мероприятий.

Страница персональных рекомендаций для учеников курсов подготовки к ЕГЭ

Это был один из первых проектов, который выполнялся в роли ученого по данным.

В компании планировали запустить новый продукт - промокурс с большими группами учеников. Наша команда должна была сделать так, чтобы качество обучения в больших группах было сравнимо с обычными группами. Для этого была разработана страница персональных достижений и рекомендаций для учеников. На которой ученик мог видеть прогресс, оценку его достижений на основе ачивок и его дальнейшие шаги в обучении. Те все то, что обычно доносит преподаватель до ученика/родителя возложено на “хрупкие плечи” страницы персональных рекомендаций.

Источник vc.ru

Создание такой страницы состояло из следующих этапов:

1️⃣ Разработка и согласование прототипа
2️⃣ Создание SQL запросов для формирования таблиц с нужными показателями
3️⃣ Разработка фронтенда страницы персональных рекомендаций
4️⃣ Создание персонализированных сообщений и их рассылка
